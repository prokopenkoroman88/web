Define your configuration files
Определите свои файлы конфигурации

Each Magento 2 module has its own set of configuration files, gathered into the module’s etc directory.
Каждый модуль Magento 2 имеет свой собственный набор файлов конфигурации, собранных в каталоге модуля etc.
Unlike Magento 1, there is no monolithic configuration file in Magento 2.
В отличие от Magento 1, в Magento 2 нет монолитного файла конфигурации.

Root directory location
Расположение корневого каталога

Use /etc for your configuration files
Используйте / etc для ваших файлов конфигурации

Magento 2 looks for configuration information for each module in that module’s etc directory. Depending on the needs of your module, you might have the following configuration files at the top level of your module’s etc directory:
Magento 2 ищет информацию о конфигурации для каждого модуля в его каталоге etc. В зависимости от потребностей вашего модуля у вас могут быть следующие файлы конфигурации на верхнем уровне каталога etc вашего модуля:
  • acl.xml
  • config.xml
  • di.xml
  • module.xml
  • webapi.xml
Additions you make to those configuration files are applied globally to your module.
Дополнения, которые вы вносите в эти файлы конфигурации, применяются к вашему модулю глобально.
In addition to those files, a Magento 2 module also has nested configuration directories in the etc directory for any required administration html, frontend, API REST, or API SOAP specific configuration. Additions you make to files in these directories override the settings in the global configuration files for the respective functionality only. That is, if you add a config.xml file to etc/frontend, the settings you make in that file overrides the settings in etc/config.xml for storefront functionality only.
В дополнение к этим файлам модуль Magento 2 также имеет вложенные каталоги конфигурации в каталоге etc для любой необходимой административной конфигурации html, внешнего интерфейса, API REST или API SOAP. Добавления, которые вы вносите в файлы в этих каталогах, отменяют настройки в глобальных файлах конфигурации только для соответствующих функций. То есть, если вы добавляете файл config.xml в etc/frontend, настройки, которые вы делаете в этом файле, переопределяют настройки в etc/config.xml только для функциональности витрины.

Configuration files
Файлы конфигурации
  • Configuration files that are in the top level of that module’s etc directory are global to that component.
  • Configuration files placed in subdirectories (adminhtml, frontend, webapi_rest, webapi_soap) apply only to those respective functional areas.
  • Файлы конфигурации, которые находятся на верхнем уровне каталога etc этого модуля, являются глобальными для этого компонента.
  • Файлы конфигурации, размещенные в подкаталогах (adminhtml, frontend, webapi_rest, webapi_soap), применяются только к соответствующим функциональным областям.
----------------------- Tailor your configuration files for what your module does The exact set of configuration files required for your module depends on what your new module does. The required configuration files depend on how you plan to use the module: will the module be manifested on the storefront UI, or in the Magento Admin panel, or as a backend extension that makes a service call? Or all of the above. For example, if your module performs a function in the Admin, you should add any necessary configuration files for those functions to etc/adminhtml/, like: /etc/adminhtml/di.xml /etc/adminhtml/routes.xml Similarly, if your module changes the UI, you should add the needed configuration files to ~/etc/frontend/. For example: /etc/frontend/di.xml /etc/frontend/page_types.xml If the module is a service that may call an API, or does some other work that is not manifested in the UI you should add any needed configuration files in the REST and/or SOAP webapi configuration directories, like this: /etc/webapi_rest/di.xml /etc/webapi_soap/di.xml Keep in mind that you might be able to handle your module’s configuration solely with configuration files at the top level of your module’s etc directory, but the nested directory is a useful way to keep the configuration neatly compartmentalized. = Адаптируйте файлы конфигурации для того, что делает ваш модуль Точный набор файлов конфигурации, необходимых для вашего модуля, зависит от того, что делает ваш новый модуль. Необходимые файлы конфигурации зависят от того, как вы планируете использовать модуль: будет ли модуль отображаться в пользовательском интерфейсе магазина, в панели администратора Magento или как внутреннее расширение, которое выполняет вызов службы? Или все, что выше. Например, если ваш модуль выполняет функцию в админке, вам следует добавить все необходимые файлы конфигурации для этих функций в etc/adminhtml/, например: <корневой каталог вашего модуля>/etc/adminhtml/di.xml <корневой каталог вашего модуля>/etc/adminhtml/routes.xml Точно так же, если ваш модуль изменяет пользовательский интерфейс, вы должны добавить необходимые файлы конфигурации в ~/etc/frontend/. Например: <корневой каталог вашего модуля>/etc/frontend/di.xml <корневой каталог вашего модуля>/etc/frontend/page_types.xml Если модуль представляет собой службу, которая может вызывать API или выполнять другую работу, которая не отображается в пользовательском интерфейсе, вам следует добавить все необходимые файлы конфигурации в каталоги конфигурации REST и / или SOAP webapi, например: <корневой каталог вашего модуля>/etc/webapi_rest/di.xml <корневой каталог вашего модуля>/etc/webapi_soap/di.xml Имейте в виду, что вы можете обрабатывать конфигурацию своего модуля исключительно с помощью файлов конфигурации на верхнем уровне каталога etc вашего модуля, но вложенный каталог - полезный способ сохранить конфигурацию аккуратно разделенной.
